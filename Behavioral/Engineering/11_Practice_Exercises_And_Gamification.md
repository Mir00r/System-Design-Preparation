## 🎮 Role-Play Scenarios

### Scenario 1: The Production Outage 🔥

**Setup**: You're an on-call Java developer. At 2 AM, PagerDuty fires: your payment service is returning 500 errors. Revenue is being lost every second.

**Your task**: Walk through EXACTLY what you would do, minute by minute.

<details>
<summary>🔑 Model Answer Framework</summary>

```
Minute 0-2: ASSESS
• Check monitoring dashboards (Grafana/Datadog)
• Identify: which service, which endpoint, error rate %
• Determine severity: P1 (all users) or P2 (partial)

Minute 2-5: COMMUNICATE  
• Post in #incidents channel: "Investigating payment service 500s"
• Page secondary on-call if P1
• Update status page for external customers

Minute 5-15: DIAGNOSE
• Check recent deployments (was something just deployed?)
• Check logs (ELK/CloudWatch) for error patterns
• Check dependencies (database, Redis, external APIs)
• Check resource metrics (CPU, memory, connections)

Minute 15-30: MITIGATE
• If recent deploy → ROLLBACK immediately
• If dependency → failover/circuit breaker
• If resource exhaustion → scale up temporarily

Minute 30+: FIX
• Implement proper fix (not just band-aid)
• Test in staging if possible
• Deploy with enhanced monitoring

After: POST-MORTEM
• Timeline of events
• Root cause analysis
• Action items to prevent recurrence
```

</details>

### Scenario 2: The Architecture Disagreement 🏗️

**Setup**: Your team is building a new notification service. The tech lead wants a synchronous REST approach. You believe an event-driven approach (Kafka) would be better for the expected 1M notifications/day. The team is split.

**Your task**: How do you handle this professionally and drive toward the best decision?

<details>
<summary>🔑 Model Answer Framework</summary>

```
Step 1: UNDERSTAND their perspective
• "Help me understand why you prefer synchronous. 
   What concerns do you have about Kafka?"
• Listen. Take notes. Find valid points.

Step 2: ESTABLISH shared criteria
• "Can we agree on what 'success' looks like?
   Throughput target? Latency SLA? Operational simplicity?"

Step 3: PRESENT data, not opinions
• Benchmark: REST at 1M/day = 12 req/sec (manageable)
• But: bursts during peak? Retry semantics? Ordering guarantees?
• Show trade-off matrix with agreed criteria

Step 4: PROPOSE experiment
• "Can we prototype both approaches this sprint?
   Let data decide, not preferences."

Step 5: COMMIT to the decision
• "Whatever we decide, I'm 100% committed to making it work."
```

</details>

---

### Scenario 3: The Underperforming Feature 📉

**Setup**: You built a recommendation engine feature 3 months ago. Metrics show it's not improving user engagement as expected. Your PM is asking what went wrong.

**Your task**: How do you respond? What do you propose?

<details>
<summary>🔑 Model Answer Framework</summary>

```
Step 1: OWN IT (not defensive)
• "Let me dig into the data and figure out what's happening."
• Don't blame the PM's requirements or the model

Step 2: ANALYZE root causes
• Is the algorithm wrong? (A/B test metrics)
• Is the placement wrong? (UX heat maps)
• Is the data quality poor? (training data analysis)
• Are we measuring the right thing? (metric alignment)

Step 3: PROPOSE iterations
• "Based on my analysis, I think [specific issue].
   Here are 3 experiments I'd like to run:
   1. Change placement (cheapest test)
   2. Adjust algorithm weights (medium effort)
   3. Expand training data (highest effort)"

Step 4: COMMIT to follow-through
• "I'll have A/B test results for option 1 by next Friday."
```

</details>

## 🎤 Mock Interview Scripts

### Mock Interview 1: General (30 minutes)

Use with a partner or record yourself:

```
INTERVIEWER SCRIPT:

[00:00] "Thanks for joining. Let's start — tell me briefly 
         about yourself and your current role."
         
[03:00] "Great. Tell me about a technically challenging 
         project you worked on recently."
         → Follow-up: "What was the hardest part?"
         → Follow-up: "What would you do differently?"
         
[10:00] "Describe a time you disagreed with a teammate 
         on a technical approach."
         → Follow-up: "How was it resolved?"
         → Follow-up: "What did you learn about collaboration?"
         
[17:00] "Tell me about a time you failed or made a mistake."
         → Follow-up: "What was the impact?"
         → Follow-up: "How did you change after that?"
         
[24:00] "Tell me about a time you went above and beyond 
         what was expected."
         → Follow-up: "Why did you take that initiative?"
         → Follow-up: "What was the outcome?"

[30:00] "Do you have any questions for me?"
```

### Mock Interview 2: Amazon-Style (45 minutes)

```
INTERVIEWER SCRIPT (Amazon Bar Raiser Style):

[00:00] "Tell me about a time you dove deep into a 
         problem to find the root cause."
         → "What data did you look at?"
         → "Who else was involved?"
         → "What alternatives did you consider?"
         → "How did you validate your hypothesis?"
         → "What was the long-term fix?"

[15:00] "Tell me about a time you made a decision 
         without complete information."
         → "What information was missing?"
         → "How did you assess the risk?"
         → "What was the outcome?"
         → "Would you make the same decision again? Why?"

[30:00] "Tell me about a time you disagreed with a 
         decision but committed to it anyway."
         → "What was your reasoning for disagreeing?"
         → "How did you express your disagreement?"
         → "How did you commit once the decision was made?"
         → "What was the final outcome?"

[45:00] End
```
