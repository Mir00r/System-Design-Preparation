### The Golden Rules 📜

```
Rule 1: 🧠 UNDERSTAND before you MEMORIZE
        → Know WHY a pattern exists, not just HOW it works

Rule 2: 🔗 CONNECT patterns to REAL problems
        → Every pattern solves a specific pain point

Rule 3: 🎯 PRACTICE with CODE, not just READING
        → Type out every example, modify, break, fix

Rule 4: 🤝 COMBINE patterns in REAL systems
        → Patterns rarely work alone in production

Rule 5: ❌ DON'T force patterns where they don't fit
        → The best pattern is sometimes NO pattern
```

## 💡 Quick Pattern Selection Guide

**"I need to..."**

| Problem | Pattern(s) to Consider |
|---------|----------------------|
| Create only ONE instance | **Singleton** |
| Create objects without specifying exact class | **Factory Method** |
| Create families of related objects | **Abstract Factory** |
| Build complex objects step-by-step | **Builder** |
| Copy existing objects efficiently | **Prototype** |
| Make incompatible interfaces work together | **Adapter** |
| Separate abstraction from implementation | **Bridge** |
| Treat individual and group objects uniformly | **Composite** |
| Add behavior dynamically without subclassing | **Decorator** |
| Simplify a complex subsystem | **Facade** |
| Share objects to save memory | **Flyweight** |
| Control access to an object | **Proxy** |
| Switch algorithms at runtime | **Strategy** |
| Notify multiple objects of state changes | **Observer** |
| Encapsulate requests as objects | **Command** |
| Define algorithm skeleton, defer steps | **Template Method** |
| Traverse collection without exposing internals | **Iterator** |
| Change behavior based on internal state | **State** |
| Pass request along a handler chain | **Chain of Responsibility** |
| Centralize complex communications | **Mediator** |
| Save and restore object state | **Memento** |
| Add operations without modifying structure | **Visitor** |
| Create a mini-language interpreter | **Interpreter** |
